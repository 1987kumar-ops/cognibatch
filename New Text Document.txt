1. Verify VM power state:
   ```bash
   az vm get-instance-view -g rg-ailab-arunkumar -n vm-app --query "instanceView.statuses[?contains(code,'PowerState')].displayStatus" -o tsv
   az vm get-instance-view -g rg-ailab-arunkumar -n vm-db --query "instanceView.statuses[?contains(code,'PowerState')].displayStatus" -o tsv
   az vm get-instance-view -g rg-ailab-arunkumar -n vm-win --query "instanceView.statuses[?contains(code,'PowerState')].displayStatus" -o tsv
   ```
2. Verify Bastion provisioning:
   ```bash
   az network bastion show -g rg-ailab-arunkumar -n bastion-ailab --query provisioningState -o tsv
   ```
3. Verify no existing critical alerts or outages in the environment.
4. Confirm there is a restore path and recovery window of 5 minutes.

## Scenario A: `vm-app` CPU exhaustion (payment service under load)

1. Description  
   Simulate sustained CPU saturation on `vm-app` to validate app resilience under compute pressure.

2. Failure simulated  
   `vm-app` CPU pegged at 100%, causing slow or failed app responses.

3. Exact trigger command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-app --command-id RunShellScript --scripts "for i in \$(seq 1 2); do nohup yes > /dev/null 2>&1 & done"
   ```

4. Expected impact
   - `vm-app`: high CPU, service response latency increases, requests may queue or fail.
   - `vm-db`: should remain available, may experience some slower request processing only if app retries.
   - `vm-win`: unaffected.
   - Bastion: unaffected.
   - NSGs/storage: unaffected.

5. Recovery command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-app --command-id RunShellScript --scripts "pkill -f '^yes$' || true"
   ```

6. Validation command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-app --command-id RunShellScript --scripts "if pgrep -f '^yes$' >/dev/null; then exit 1; fi"
   ```

7. RTO target  
   5 minutes

## Scenario B: DB connection pool exhaustion (`max_connections` reached)

1. Description  
   Simulate PostgreSQL reaching `max_connections=20` on `vm-db` and confirm app behavior under exhausted DB sessions.

2. Failure simulated  
   New database connections rejected with `too many connections`.

3. Exact trigger command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-db --command-id RunShellScript --scripts "for i in \$(seq 1 20); do nohup sudo -u postgres psql -d labdb -c 'select pg_sleep(300);' > /tmp/db_conn_test_\$i.log 2>&1 & done"
   ```

4. Expected impact
   - `vm-db`: accepts existing connections, rejects new ones.
   - `vm-app`: database errors should appear for new requests, likely service errors or timeouts.
   - `vm-win`: unaffected.
   - Bastion: unaffected.
   - Storage: unaffected.

5. Recovery command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-db --command-id RunShellScript --scripts "pkill -f 'select pg_sleep(300)' || true"
   ```

6. Validation command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-db --command-id RunShellScript --scripts "sudo -u postgres psql -d labdb -c 'select 1'"
   ```

7. RTO target  
   5 minutes

## Scenario C: Disk fill on `vm-app` (production write failures)

1. Description  
   Simulate root filesystem exhaustion on `vm-app` to validate app behavior when writes fail.

2. Failure simulated  
   App cannot write logs/temp files; write errors occur.

3. Exact trigger command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-app --command-id RunShellScript --scripts "sudo fallocate -l 24G /tmp/vm_app_disk_fill_test || sudo dd if=/dev/zero of=/tmp/vm_app_disk_fill_test bs=1M count=24000"
   ```

4. Expected impact
   - `vm-app`: write operations fail, service may show 500s or disk full errors.
   - `vm-db`: unaffected.
   - `vm-win`: unaffected.
   - Bastion: unaffected.
   - Storage: unaffected.

5. Recovery command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-app --command-id RunShellScript --scripts "rm -f /tmp/vm_app_disk_fill_test"
   ```

6. Validation command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-app --command-id RunShellScript --scripts "df -h /tmp | awk 'NR==2 {print \$5}'"
   ```

7. RTO target  
   5 minutes

## Scenario D: Network routing misconfiguration (`app` cannot reach `db`)

1. Description  
   Simulate a broken route/firewall on `vm-app` that blocks traffic to the DB IP.

2. Failure simulated  
   `vm-app` loses connectivity to `vm-db` at `10.0.2.10`.

3. Exact trigger command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-app --command-id RunShellScript --scripts "sudo iptables -A OUTPUT -d 10.0.2.10 -j DROP"
   ```

4. Expected impact
   - `vm-app`: cannot establish DB connections, app errors on database access.
   - `vm-db`: remains healthy and reachable from other hosts.
   - `vm-win`: unaffected.
   - Bastion: unaffected.
   - NSGs/storage: unchanged.

5. Recovery command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-app --command-id RunShellScript --scripts "sudo iptables -D OUTPUT -d 10.0.2.10 -j DROP || true"
   ```

6. Validation command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-app --command-id RunShellScript --scripts "bash -c 'if timeout 5 bash -c \"cat < /dev/tcp/10.0.2.10/5432\" >/dev/null 2>&1; then echo CONNECTED; else echo BLOCKED; fi'"
   ```

7. RTO target  
   5 minutes

## Scenario E: Windows IIS service failure (`vm-win` reporting service down)

1. Description  
   Simulate the Windows reporting service shutdown by stopping IIS on `vm-win`.

2. Failure simulated  
   `W3SVC` (IIS) service stops, making reporting/UI unavailable.

3. Exact trigger command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-win --command-id RunPowerShellScript --scripts "Stop-Service W3SVC -Force"
   ```

4. Expected impact
   - `vm-win`: IIS is down, reporting/service web endpoint unavailable.
   - `vm-app`: unaffected unless it depends on `vm-win`.
   - `vm-db`: unaffected.
   - Bastion: unaffected.
   - Storage: unaffected.

5. Recovery command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-win --command-id RunPowerShellScript --scripts "Start-Service W3SVC"
   ```

6. Validation command
   ```bash
   az vm run-command invoke -g rg-ailab-arunkumar -n vm-win --command-id RunPowerShellScript --scripts "Get-Service W3SVC | Select-Object -ExpandProperty Status"
   ```

7. RTO target  
   5 minutes

4) **How dropped connection causes deadlock on reconnect**

**Chain of events (using alert data):**

1. At 09:14:47 (ALT-007), `vm-app` first attempts DB connection to 10.0.2.10:5432 — times out (likely due to NSG rule change).

3. At 09:15:28 (ALT-021), PID 12346 is waiting for a ShareLock held by PID 12341.
4. At 09:15:44 (ALT-025), deadlock detected between PID 12341 and 12342, both waiting for ShareLock — rolled back.

**Why the deadlock occurs after the drop:**
- When the client drops uncleanly, the transaction on PID 12341/12342 is left in an uncommitted state (locks still held on the database until cleanup).
- Cleanup is delayed or the connection slot persists.
- When `vm-app` retries the same logical transaction, it reconnects and the backend tries to re-acquire locks.
- If the app sends conflicting lock requests in different orders on the retry (e.g., lock A then B on one PID, lock B then A on another), a circular wait forms → deadlock.
- PostgreSQL detects the cycle and rolls back one transaction, but the slot remains occupied (ALT-025).

5) **SQL to safely terminate idle-in-transaction connections**

```sql
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE state = 'idle in transaction'
  AND state_change < NOW() - INTERVAL '5 minutes'
  AND pid != pg_backend_pid();
```

**Safer version (only kill obvious hangers, not recent ones):**
```sql
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE state = 'idle in transaction'
  AND (NOW() - state_change) > INTERVAL '30 seconds'
  AND backend_type = 'client backend'
  AND pid != pg_backend_pid();
```

**Why safe:** only terminates idle-in-transaction (not active queries), excludes the admin connection (`pg_backend_pid()`), and allows a grace period before killing.

7) **Minimum PgBouncer configuration to prevent exhaustion**

PgBouncer acts as a connection pooler, multiplexing many client connections into fewer backend connections to PostgreSQL.

**pgbouncer.ini:**
```ini
[databases]
labdb = host=10.0.2.10 port=5432 dbname=labdb user=labuser password=Lab@2024!

[pgbouncer]
logfile = /var/log/pgbouncer/pgbouncer.log
pidfile = /var/run/pgbouncer/pgbouncer.pid

; Client-facing settings
listen_addr = 0.0.0.0
listen_port = 6432
max_client_conn = 100

; Backend pool settings (PostgreSQL connections)
default_pool_size = 8
min_pool_size = 4
reserve_pool_size = 2
reserve_pool_timeout = 3

; Transaction pooling: release connection after each transaction (fastest safe mode)
pool_mode = transaction

; Connection reuse and cleanup
connection_lifetime = 3600
idle_in_transaction_session_timeout = 60000

; TCP tuning
tcp_keepalives = 1
tcp_keepidles = 5
tcp_keepintvls = 5
```

**Why this prevents exhaustion:**

| Parameter | Value | Purpose |
|-----------|-------|---------|
| `max_client_conn` | 100 | PgBouncer accepts up to 100 client connections |
| `default_pool_size` | 8 | But only maintains 8 actual PostgreSQL connections (vs 20 max) |
| `min_pool_size` | 4 | Minimum 4 connections kept open, ready for use |
| `reserve_pool_size` | 2 | Emergency 2 extra connections for spikes |
| `pool_mode = transaction` | – | Release connection back to pool after each transaction (not after idle time) |

**Result:** app can queue 100 client connections, but only 8–10 actual DB connections used, preventing exhaustion of the `max_connections=20` limit on PostgreSQL. Excess app connections wait in PgBouncer's queue briefly and are served as DB connections become available.

2. **ALT-024 IOPS threshold appropriateness and recommendation**

**Current threshold:** 500 IOPS  
**Is it appropriate?** No — it fires during a transient error log spike, which is not indicative of a storage problem.

**Why?**
- Standard_B2ms disk performance is designed to handle sustained reads/writes well above 500 IOPS.
- 840 IOPS is expected during an application incident with high-volume error logging.
- The alert does not distinguish between healthy load and unhealthy load.

**Recommended change:**
Raise threshold to **2000–3000 IOPS** and add a correlation condition:
```
Alert fires if (IOPS > 2000) AND (error_rate > 20% OR connection_timeouts > 10/min)
```

**Rationale:** This eliminates noise from transient retries and only alerts when disk stress correlates with a genuine application problem that requires storage-level intervention (e.g., query plan explosion, checkpoint storm).
